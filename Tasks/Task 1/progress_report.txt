Bugs and Challenges Faced

1. XQuartz display variable issue

Problem:

Running:

xhost +localhost

failed with:

xhost: unable to open display "host.docker.internal:0"

Cause:

The DISPLAY variable was set to:

host.docker.internal:0

That value is meant for Docker containers to reach the host display. The xhost command runs on the host side and needs to talk directly to XQuartz, so it should use:

:0

Solution:

Use this order:

export DISPLAY=:0
xhost +localhost
export DISPLAY=host.docker.internal:0

The first DISPLAY value is for the host-side XQuartz permission command. The second DISPLAY value is for Docker GUI containers.


2. Gazebo and RViz2 OpenGL rendering failure

Problem:

The Docker container started, but Gazebo and RViz2 failed with OpenGL errors.

Observed errors included:

Failed to create OpenGL context
Unable to create a suitable GLXContext
Failed to create the rendering window

Cause:

ROS 2, Gazebo, and RViz2 were running inside Docker, but their GUI windows needed OpenGL rendering through XQuartz. On macOS, Docker containers do not have direct access to the native graphics stack. XQuartz provides the display bridge, but OpenGL/GLX support through this path can be limited or fragile, especially for Gazebo and RViz2.

This is a graphics/display integration issue, not a ROS build issue.

Evidence:

The ROS launch logs showed that the map server, AMCL, Nav2 lifecycle nodes, and other ROS components started before the GUI rendering failure. That means the environment was built and launched, but the GUI rendering layer failed.

Potential solution:

Retry the launch with software rendering:

LIBGL_ALWAYS_SOFTWARE=1 \
MESA_GL_VERSION_OVERRIDE=3.3 \
MESA_GLSL_VERSION_OVERRIDE=330 \
QT_XCB_GL_INTEGRATION=none \
docker compose up demo-world

Alternate option:

LIBGL_ALWAYS_INDIRECT=1 \
QT_XCB_GL_INTEGRATION=none \
docker compose up demo-world


Why These Challenges Happened

1. Docker isolates the Linux ROS environment from the host operating system.

This is useful because ROS and Gazebo are easier to run in Linux than directly on macOS, but it creates extra complexity for GUI applications.

2. XQuartz is required because Gazebo and RViz2 are Linux GUI applications.

Without XQuartz, the containers have no display server to connect to.

3. DISPLAY has two different roles.

For xhost on the host side:

DISPLAY=:0

For Docker containers:

DISPLAY=host.docker.internal:0

Using the Docker display value for xhost causes xhost to fail.

4. Gazebo and RViz2 require OpenGL.

Simple GUI forwarding is not always enough. These tools need a working OpenGL context, and that is the part currently failing through XQuartz.
